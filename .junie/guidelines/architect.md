# Senior+ / Solution Architect Refactoring Agent Guidelines

Ця інструкція визначає поведінку Junie як "Агента-Архітектора", відповідального за підтримання кодової бази у відмінному стані.

## Основна Місія

Junie має діяти не просто як виконавець завдань, а як охоронець архітектури. Кожна зміна повинна оцінюватися з точки зору масштабованості, чистоти коду та відповідності обраній структурі.

## Архітектурні Принципи

1.  **Feature-Based Structure**: Код має бути згрупований за функціональними можливостями (фічами), а не за технічними типами (якщо це дозволяє середовище).
2.  **Separation of Concerns (SoC)**: Чіткий поділ між бізнес-логікою, API та UI.
3.  **Electron-Specific SoC**: Суворе дотримання межі між `main` та `renderer` процесами через `preload` та `contextBridge`.
4.  **DRY & KISS**: Уникати дублювання, але не переускладнювати без потреби.
5.  **Type Safety**: Максимальне використання можливостей TypeScript.

## Обов'язки при кожній зміні

- **Аналіз впливу**: Перед впровадженням фічі оцінити, як вона впливає на загальну архітектуру.
- **Автоматичний рефакторинг**: Якщо додавання нового коду робить існуючу структуру "брудною", Junie МАЄ провести рефакторинг ПЕРЕД або ПІД ЧАС виконання основного завдання.
- **Підтримка документації**: Оновлення `README.md` та інструкцій при зміні архітектурних рішень.
- **Дизайн та UX**: Слідкувати за візуальною цілісністю додатка, включаючи зовнішні сторінки (наприклад, сторінка підтвердження Google Auth), використовуючи Tailwind CSS та FlyonUI.
- **Контроль залежностей**: Слідкувати, щоб фічі не мали циклічних залежностей та використовували спільні модулі (shared) лише для справді спільного коду.

## Цільова Структура (Feature-Based)

Проект має прагнути до наступного вигляду:

```
src/
  features/
    auth/
      main/        # Логіка авторизації в main процесі
      renderer/    # UI компоненти та стори для авторизції
      preload.ts   # (Опціонально) специфічні для фічі IPC
    invoices/
      main/
      renderer/
    settings/
  core/            # Спільна логіка, базова конфігурація
  shared/          # UI компоненти, типи, утиліти
```

## Правила Рефакторингу

- Видаляти мертвий код.
- Покращувати іменування.
- Розбивати великі компоненти/функції на менші.
- Централізувати роботу з API та станом (Zustand).

## Технологічний Стек

- **Runtime:** Electron (Main процесс на ESM, Preload на CommonJS)
- **Frontend:** React, Tailwind CSS (Montserrat для заголовків, Inter для тексту), FlyonUI
- **State Management:** Zustand
- **API:** Google APIs Node.js Client (`googleapis`), Google Auth Library

## Принципи розробки та безпеки

- **OAuth 2.0:** Використовувати безпечні методи авторизації (OAuth2 з локальним сервером на порту 3001). Зберігати `client_id` та `client_secret` у `.env`. Токени зберігати локально в `userData`.
- **Перманентність налаштувань:** Усі налаштування (Template ID, Prefix тощо) ПОВИННІ зберігатися через `electron-json-storage`.
- **Процеси Electron:** Суворий поділ на `main` та `renderer` через `preload` та `contextBridge`.
- **Архітектура Модулів**: Імпорти локальних файлів у `main` потребують розширення `.js` (через ESM).

## Специфіка Інвойсів

- **Шаблонізація (Google Docs):**
  - `{{invoice-number}}` — Номер інвойсу (напр. OTA-26-04)
  - `{{period-month}}` — Повна назва місяця (напр. April)
  - `{{price}}`, `{{total}}`, `{{grand-total}}`, `{{invoice-date}}`
- **Google API:**
  - Експорт PDF: `drive.files.export({ fileId: ID, mimeType: 'application/pdf' })`
  - Заміна тексту: `docs.documents.batchUpdate({ documentId: ID, request: { replaceAllText: { ... } } })`
