# Electron Invoicer App

Додаток для створення інвойсів з інтеграцією Google Docs, Google Drive та Google Forms. Побудований на Electron, React, TypeScript, Tailwind CSS, FlyonUI та Zustand.

## 1. Локальна розробка

### Попередні вимоги

- Node.js (рекомендовано LTS)
- npm

### Налаштування Google API

1. Створіть проект у [Google Cloud Console](https://console.cloud.google.com/).
2. Увімкніть **Google Docs API**, **Google Drive API** та переконайтеся, що OAuth2 дозволяє запити профілю.
3. Налаштуйте **OAuth Consent Screen** (External, додайте свій email у Test Users). Додайте області (scopes): `userinfo.profile` та `userinfo.email`.
4. Створіть **OAuth 2.0 Client ID** (тип: Web Application).
5. Додайте `http://localhost:3001` в **Authorized redirect URIs**.
6. Скопіюйте `Client ID` та `Client Secret` у файл `.env`.

### Змінні середовища (.env)

```env
GOOGLE_CLIENT_ID=ваш_id.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=ваш_секрет
```

## 2. Робота з шаблонами

Програма використовує Google Docs як шаблони. В тексті документа можна використовувати наступні плейсхолдери:

- `{{invoice-number}}` — Номер інвойсу (напр. OTA-26-04). Префікс (OTA) налаштовується в параметрах додатка.
- `{{period-month}}` — Повна назва місяця (напр. April)
- `{{price}}` — Сума інвойсу з валютою (напр. €3563.89)
- `{{total}}` — Сума інвойсу (наразі співпадає з price)
- `{{grand-total}}` — Загальна сума (наразі співпадає з price)
- `{{invoice-date}}` — Дата інвойсу (останній день місяця)

**Як отримати Template ID:** Відкрийте ваш шаблон у браузері, ID знаходиться в URL: `docs.google.com/document/d/ID_ТУТ/edit`. В поле налаштувань можна вставити як ID, так і повний URL.

## 3. Налаштування

В додатку (іконка шестерні) можна налаштувати:

- **Префікс інвойсу**: Рядок, з якого починається номер інвойсу (напр. `OTA`).
- **Google Doc Template ID**: ID або URL вашого шаблону.
- **Папка в Google Drive**: Назва папки, де будуть автоматично створюватися копії інвойсів (за замовчуванням `invoices`).
  Налаштування зберігаються перманентно.

Для тестування версії, максимально наближеної до продуктової (використання зібраних файлів замість dev-сервера):

1. Зберіть проект:
   ```bash
   npm run build
   ```
2. Запустіть Electron:
   ```bash
   npm start
   ```
   _Примітка: Переконайтеся, що в `.env` не встановлено `NODE_ENV=development` або він встановлений у `production`, щоб додаток завантажував локальні файли з папки `dist`._

Для створення готових до розповсюдження пакетів використовується `electron-builder` (необхідно встановити та налаштувати для фінальної збірки) або базові скрипти.

## 4. GitHub Actions та Secrets

Проект налаштований на автоматичну збірку релізів. Для коректної роботи необхідно додати наступні **Secrets** у ваш GitHub репозиторій (`Settings -> Secrets and variables -> Actions`):

1. `GOOGLE_CLIENT_ID`: Ваш Google Client ID.
2. `GOOGLE_CLIENT_SECRET`: Ваш Google Client Secret.
3. `GITHUB_TOKEN`: (Зазвичай надається автоматично, але переконайтеся, що Actions мають права `write` на вміст репозиторію).

Збірка запускається автоматично при створенні тегу версії (наприклад, `v1.0.1`).

## 5. Архітектура

Проект використовує **Feature-Based Architecture**. Кожна функціональна частина додатку (Auth, Invoices, Settings) має свою власну папку, що містить логіку для обох процесів (main та renderer).

### Структура `src/`

- `features/`: Функціональні модулі додатку.
  - `auth/`: Авторизація через Google.
  - `invoices/`: Генерація та робота з інвойсами.
  - `settings/`: Керування налаштуваннями додатку.
- `core/`: Глобальна конфігурація, ініціалізація Electron.
- `shared/`: Спільні компоненти, типи, утиліти та глобальні стори (Zustand).
- `preload/`: Скрипти для `contextBridge`.

### Агент-Архітектор (Refactoring Agent)

У проекті задіяно роль **Senior+/Solution Architect Agent**, який:

- Слідкує за чистотою кодової бази та дотриманням архітектурних принципів.
- Проводить автоматичний рефакторинг при розширенні функціоналу.
- Підтримує документацію в актуальному стані.
- Правила роботи агента описані в `.junie/guidelines/architect.md`.

## Важливі зауваження

- **ESM в Main Process**: Проект використовує `"type": "module"`. Завдяки налаштуванням TypeScript, імпорти виконуються без розширень `.js`.
- **CommonJS в Preload**: Preload-скрипт компілюється в CommonJS для забезпечення сумісності з Electron.
- Всі локальні імпорти мають використовувати @-аліаси (наприклад, `@renderer/...`, `@features/...`).
- Завжди оновлюйте `.env` новими ключами при зміні середовища.
- Всі запити до Google APIs проходять через мейн-процес для забезпечення безпеки токенів.
